# 动态端口显示与统一状态链路测试文档

## 1. 文档目的

这份文档不是讲方案，而是给当前这轮联调一个可执行的测试标准。

它要回答 4 个问题：

- 这轮要不要重烧 `ESP32`
- 这轮到底测什么
- 屏幕、网页、硬件各自该看到什么
- 什么现象算通过，什么现象直接判失败

## 3. 这轮是否需要重烧 ESP32

### 3.1 结论

这轮屏幕“动态模块入口显示”的改动，`ESP32` 不需要因为 UI 改动而单独重烧。

### 3.2 前提

上面这个结论成立，必须同时满足以下条件：

1. 你现在板子上的 `ESP32` 已经是这套桥接固件：
   - [esp32_s3_uart_link.ino](D:/Renesas-Workspace/RA8P1_Viewing_Screen_ESP32_Link/esp32-s3-uart-link-arduino/esp32_s3_uart_link/esp32_s3_uart_link.ino)
2. `ESP32` 当前已经能正常：
   - 连 WiFi
   - 连 MQTT
   - 转发 `status / telemetry`
3. 网页或原始状态页已经能看到设备在线

### 3.3 什么时候才需要重烧 ESP32

只有下面几种情况，才需要重新烧 `ESP32`：

- 当前 `ESP32` 不是上面这版桥接固件
- `UART` 桥接异常
- `WiFi / MQTT` 连不上
- 网页侧完全收不到 `status / telemetry`
- 你后面继续改了 `ESP32` 的协议解析或上报逻辑

### 3.4 本轮必须重烧什么

本轮必须重烧的是 `RA8P1`，因为这次改动实际发生在屏幕/UI 和板端统一状态消费逻辑：

- [RA8P1_Viewing_Screen_ESP32_Link.srec](D:/Renesas-Workspace/RA8P1_Viewing_Screen_ESP32_Link/Debug/RA8P1_Viewing_Screen_ESP32_Link.srec)

## 4. 本轮测试要验证的事实

本轮不是验证“有没有做一个 BH1750 页面”，而是验证下面这些事实：

1. `i2c.s1` 这个真实物理标准口可以承载多个模块
2. `AHT20` 和 `BH1750` 可以通过 `PCA9548A` 同时接入
3. 屏幕“接入硬件”页按真实接入结果动态显示多个模块入口
4. 网页侧仍然保持真实物理口语义，不把 `BH1750` 伪造成 `i2c.s2`
5. 网页、服务器、ESP32、RA8P1、屏幕看到的是同一条状态链

## 5. 当前正确接线

### 5.1 RA8P1 到 PCA9548A

| RA8P1 | PCA9548A | 说明 |
|---|---|---|
| `P309` | `SDA` | `I2C-1 SDA` |
| `P306` | `SCL` | `I2C-1 SCL` |
| `3V3` | `VIN` | 3.3V 供电 |
| `GND` | `GND` | 共地 |

### 5.2 PCA9548A 到 AHT20

| PCA9548A | AHT20 | 说明 |
|---|---|---|
| `SD0` | `SDA` | 子通道 0 |
| `SC0` | `SCL` | 子通道 0 |
| `3V3` | `VCC` | 3.3V 供电 |
| `GND` | `GND` | 共地 |

### 5.3 PCA9548A 到 BH1750

| PCA9548A | BH1750 | 说明 |
|---|---|---|
| `SD1` | `SDA` | 子通道 1 |
| `SC1` | `SCL` | 子通道 1 |
| `3V3` | `VCC` | 3.3V 供电 |
| `GND` | `GND` | 共地 |

### 5.4 接线红线

下面这些情况直接视为接线不合格：

- `AHT20` 仍然直连在 `P309/P306`
- `BH1750` 和 `AHT20` 直接并到 `P309/P306`
- `AHT20` 没走 `CH0`
- `BH1750` 没走 `CH1`
- `PCA9548A`、`AHT20`、`BH1750` 没有共地

## 8. 屏幕端预期现象

### 8.1 接入硬件页

在当前接线下，屏幕“接入硬件”页至少应出现 2 个动态模块入口：

- `温湿度模块`
- `光照模块`

副信息应能体现具体型号，例如：

- `AHT20`
- `BH1750`

如果以后同一标准口下再挂别的模块，入口数量应该继续增加，而不是把现有页面覆盖掉。

### 8.2 I2C-1 扫描

当前设备只有 `PCA9548A + AHT20 + BH1750` 时，屏幕上应能看到以下扫描结果：

- `0x70/9548A-MUX`
- `0x38/AHT20`
- `0x23/BH1750`

如果 `BH1750 ADDR` 脚导致地址切换，也允许看到：

- `0x5C/BH1750`

### 8.3 模块详情页

在当前实现下：

- `AHT20` 详情应能看到真实温度、湿度
- `BH1750` 详情应能看到真实 `lux`
- 状态不应长期停留在 `offline`

---

## 9. 网页端预期现象

### 9.1 标准口语义

网页侧必须保持真实口语义：

- `i2c.s1` 是当前唯一真实 I2C 标准口
- `i2c.s2` 仍然应是 `reserved`
- `BH1750` 不能被网页错显示成 `i2c.s2`

### 9.2 capability

`i2c.s1` 当前至少应承载以下能力：

- `env.temperature`
- `env.humidity`
- `env.light.lux`

### 9.3 samples

网页侧样本至少应出现：

- `AHT20 -> env.temperature`
- `AHT20 -> env.humidity`
- `BH1750 -> env.light.lux`

## 10. 测试步骤与通过标准

## 10.1 步骤 A：基础上线

重新上电后，等待 `20` 秒内，检查：

1. 屏幕亮起
2. `ESP32` WiFi/MQTT 恢复在线
3. 网页能看到设备在线

### 通过条件

- 三项全部满足

### 失败条件

- 屏幕不启动
- `ESP32` 未联网
- 网页设备不在线

## 10.2 步骤 B：屏幕动态入口检查

进入屏幕“接入硬件”页。

### 通过条件

满足以下全部条件：

1. 页面不是固定写死的单设备页
2. 当前至少出现 `温湿度模块` 和 `光照模块` 两个入口
3. 副信息能区分 `AHT20` 和 `BH1750`

### 失败条件

- 只出现一个固定硬件入口
- `BH1750` 不出现
- 插了两个模块，屏幕却只展示一个模块

## 10.3 步骤 C：屏幕扫描检查

查看 `I2C-1` 扫描信息。

### 通过条件

满足以下全部条件：

1. 有 `0x70/9548A-MUX`
2. 有 `AHT20`
3. 有 `BH1750`
4. 三者都出现在 `I2C-1` 范围内

### 失败条件

- 只看到复用器，看不到下游模块
- 只看到 `AHT20`，看不到 `BH1750`
- `BH1750` 出现在假想的第二物理 I2C 口上

## 10.4 步骤 D：屏幕数值检查

分别进入 `AHT20` 和 `BH1750` 详情。

### 通过条件

- `AHT20` 为 `online`
- `BH1750` 为 `online`
- `20` 秒内出现有效温湿度
- `20` 秒内出现有效 `lux`

### 失败条件

- 任一模块长期 `offline`
- 温湿度或 `lux` 长期为空
- 模块入口存在，但详情始终没有真实值

## 10.5 步骤 E：网页标准口结构检查

查看网页的 `status.ports`。

### 通过条件

- `i2c.s1` 正常在线并承载当前模块
- `i2c.s2` 没被错误激活
- 网页没有把 `BH1750` 当成第二个真实 I2C 口

### 失败条件

- 出现假的 `i2c.s2 online`
- `BH1750` 被错误映射到 `i2c.s2`

## 10.6 步骤 F：网页样本检查

查看 `telemetry.samples`。

### 通过条件

- 有 `env.temperature`
- 有 `env.humidity`
- 有 `env.light.lux`
- 三个值都不是长期空值

### 失败条件

- 只有模块名字，没有样本值
- 只有 `AHT20` 数据，没有 `lux`
- 只有 `BH1750` 名字，没有 `env.light.lux`

## 10.7 步骤 G：BH1750 动态响应检查

对 `BH1750` 做两次环境变化：

1. 用手遮挡
2. 用手机手电筒照射

### 通过条件

满足以下任一项即可：

1. 遮挡和照射之间的 `lux` 差值 `>= 50`
2. 相对变化幅度 `>= 30%`

并且每次变化后 `15` 秒内网页或屏幕应看到数值变化。

### 失败条件

- `lux` 长时间不变
- 遮挡和照射后没有明显方向性变化

## 11. 最终验收判定

只有下面 7 步全部通过，才算本轮测试通过：

1. 步骤 A 通过

3. 步骤 C 通过
4. 步骤 D 通过

## 13. 本文档给出的直接结论

当前这轮如果你已经烧过前面那版 `ESP32` 桥接固件，而且它现在联网和上报都正常，那么：

- `ESP32` 不需要因为这次屏幕动态入口改动而重烧
- `RA8P1` 需要使用最新 `srec`
- 测试重点应该放在：
  - 屏幕是否动态显示多个模块
  - 网页是否仍然保持真实 `i2c.s1` 语义
  - `lux / 温度 / 湿度` 是否都能从同一链路读出来
