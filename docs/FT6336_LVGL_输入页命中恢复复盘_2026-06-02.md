# FT6336 + LVGL 输入页命中恢复复盘

日期：2026-06-02

## 问题现象

- 触摸计数一直增加，说明触摸芯片和主控通信基本正常。
- `INPUT` 按钮多次点击后，页面仍停留在首页，没有进入输入页。
- 屏幕上原来的 `TOUCH:` 数值一直跳变，但这并不能直接证明按钮命中正常。

## 最初的误判

前面的排查里有两个误判：

1. 把屏幕上的 `TOUCH:` 文本当成了 LVGL 实际使用的屏幕坐标。
实际不是。早期这行显示的是原始触摸值或不完整中间值，只能说明“有触摸”，不能说明“命中按钮”。

2. 过早把问题归因到按钮事件类型。
把 `CLICKED` 改成 `PRESSED` 有帮助，但这不是核心根因。核心仍然是触摸点到 LVGL 对象的命中链是否正确。

## 真实链路

这条输入链路可以拆成 4 层：

1. `FT6336`
读取原始触摸点 `x/y/touches/event/id`

2. `lv_port_indev.c`
把原始点映射成 LVGL 使用的绝对屏幕坐标，并在 `read_cb` 中上报

3. `LVGL display`
默认显示分辨率是 `320 x 480`

4. `app_ui.c`
按钮对象、页面切换、键盘和输入框都在这一层

只要第 2 层的坐标不对，第 4 层的按钮就会“看得见但点不到”。

## 真正起作用的修复

### 1. 触摸映射改成绝对屏幕坐标

位置：
- [src/lv_port_indev.c](D:/Renesas-Workspace/RA8P1_Viewing_Screen_ESP32_Link/src/lv_port_indev.c)

做法：
- 使用校准区间：
  - `TOUCH_RAW_X_MIN = 28`
  - `TOUCH_RAW_X_MAX = 287`
  - `TOUCH_RAW_Y_MIN = 56`
  - `TOUCH_RAW_Y_MAX = 432`
- 先把原始点映射到“可触摸区域”
- 再转换成 LVGL 需要的绝对屏幕坐标

关键点：
- 屏幕总高度是 `480`
- 顶部状态栏高度是 `32`
- 实际主页面区域从 `y = 32` 开始

这次最关键的修正就是：
- 不再把触摸直接映射到 `0..479`
- 而是先映射到内容区高度，再补上 `TOUCH_UI_TOP_OFFSET = 32`

如果不补这个偏移，触摸点会整体上移，表现就是：
- 手指看起来点到了按钮
- LVGL 实际认为点在按钮上方

### 2. 加轻量 release 去抖

位置：
- [src/lv_port_indev.c](D:/Renesas-Workspace/RA8P1_Viewing_Screen_ESP32_Link/src/lv_port_indev.c)

做法：
- 加入 `TOUCH_RELEASE_DEBOUNCE = 2`
- 触摸刚松开时，不立刻把状态切成完全释放
- 给 `PRESSED -> RELEASED` 一个很短的缓冲

目的：
- 减少触摸面板抬手瞬间抖动导致的对象丢失
- 让 LVGL 更容易稳定完成一次点击判定

### 3. 按钮导航从 `CLICKED` 改成 `PRESSED`

位置：
- [src/app_ui.c](D:/Renesas-Workspace/RA8P1_Viewing_Screen_ESP32_Link/src/app_ui.c)

做法：
- `lv_obj_add_event_cb(button, cb, LV_EVENT_PRESSED, NULL);`

原因：
- `CLICKED` 依赖一次完整的 `press -> release -> 仍命中对象`
- 当前这块触摸屏在 release 阶段更容易抖动
- 输入页这种“页面跳转型按钮”更适合按下即切页

这一步不是核心根因，但它提高了稳定性。

### 4. 扩大 `INPUT` 按钮点击区

位置：
- [src/app_ui.c](D:/Renesas-Workspace/RA8P1_Viewing_Screen_ESP32_Link/src/app_ui.c)

做法：
- `lv_obj_set_ext_click_area(button, 18);`

作用：
- 扩大按钮外侧可点击热区
- 对粗校准、手指点击、边缘点击更宽容

### 5. 加“命中探针”而不是继续猜

位置：
- [src/app_ui.h](D:/Renesas-Workspace/RA8P1_Viewing_Screen_ESP32_Link/src/app_ui.h)
- [src/app_ui.c](D:/Renesas-Workspace/RA8P1_Viewing_Screen_ESP32_Link/src/app_ui.c)
- [src/lv_port_indev.c](D:/Renesas-Workspace/RA8P1_Viewing_Screen_ESP32_Link/src/lv_port_indev.c)

做法：
- 暴露 `app_ui_touch_hits_input(int32_t x, int32_t y)`
- 在 `read_cb` 里直接判断当前映射点是否命中 `INPUT` 按钮的点击区域
- 再把结果显示为：
  - `input hit`
  - `input miss`
  - `input release`

这一步很重要，因为它把问题从：
- “为什么点了没反应”

变成了：
- “点到底有没有命中按钮”

这让排查从猜测变成了可观测。

## 为什么最后成功了

这次成功不是单一改动造成的，而是 4 个修正叠加：

1. 坐标映射到正确的 LVGL 绝对坐标
2. 补齐顶部状态栏偏移
3. release 去抖
4. 导航按钮提前在 `PRESSED` 阶段执行

其中最关键的是前两项。

## 如何复用这套技能

以后遇到“触摸能读到，但按钮点不开”的情况，不要直接改 UI 布局，按这个顺序做：

1. 先确认触摸芯片确实有稳定原始点
- 先验证 `touches/x/y/event`
- 不要先看按钮

2. 确认显示分辨率和 UI 实际可交互区域
- 尤其注意：
  - 顶部状态栏
  - 安全区
  - 页面容器偏移

3. 把原始点映射成 LVGL 使用的“绝对屏幕坐标”
- 不要只映射到内容区坐标后就直接上报

4. 给 `read_cb` 加轻量 release 去抖
- 不要让触摸刚抬手就立刻抖成 release

5. 关键按钮先用 `LV_EVENT_PRESSED`
- 特别适合：
  - 页面切换
  - 弹窗打开
  - 模式切换

6. 给目标按钮加扩展点击区
- 对小屏、手指输入、粗校准尤其有效

7. 一定加“命中探针”
- 直接判断“映射点是否命中目标对象”
- 不要只靠屏幕坐标文本猜
