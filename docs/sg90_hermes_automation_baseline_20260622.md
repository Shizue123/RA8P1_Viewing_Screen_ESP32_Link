# SG90、Hermes 与多传感器任务基线（2026-06-22）

## 已实现架构

自然语言入口统一为：

`Web / QQBot -> Hermes/模型理解上下文与最新上报 -> 服务端结构校验 -> 即时动作或持久任务 -> MQTT -> ESP32 -> RA8P1 -> SG90(P105)`

- SG90 仍使用已经验证的 `manual_action.v1` 下发链路，不修改板端 `rule_program.v1` 和既有 ACK/执行状态协议。
- 即时动作支持 `both / left / right`，旧指令不写方向时保持原来的左右往复行为。
- 温度、湿度、光照规则存入云端 SQLite，可并行存在，不再受板端单条温度规则限制。
- 条件判断只读取服务器刚收到的设备数据；条件从“不满足”变为“满足”时触发，避免每秒重复驱动舵机。
- 一次性与每日汇报任务支持 Web 会话和 QQBot 主动消息。

## 支持示例

- `当温度达到32度时，舵机向左和向右各转动2次，30度`
- `当湿度达到60%时，舵机持续向左转动2次，60度`
- `当光照低于50lux时，舵机持续向右转动3次，30度`
- `晚上九点十分时向我汇报温湿度情况和光照情况`
- `每天早上八点向我汇报温湿度和光照情况`
- `查看任务`
- `取消 task_xxxxxxxxxxxx`

## 任务周期与对话生命周期

- 只给出时间而没有说明周期时，Hermes/服务端不会直接建任务，而会追问“每天还是仅今天一次”。
- `今天 / 今晚 / 短期任务 / 仅一次` 建立一次性任务。
- `每天 / 每日 / 长期 / 永久` 建立长期任务。
- 删除 Web 对话时，该对话产生的普通任务会一并停用；只有明确表达长期存在意图的任务保留。
- 对话内的舵机自动复位偏好也会随对话删除。

## 舵机保持策略

- 手动动作默认保持最后目标角度，不再隐式回到 90 度。
- 明确说“自动复位 / 回中 / 回到 90 度”时，本次动作末尾追加 90 度。
- 明确说“以后都自动复位”时，只在当前对话保存该偏好；后续可用“以后都不要自动复位”取消。

## 完整时间

- ESP32 从 NTP 获取 `Asia/Shanghai` 的年月日、时分秒。
- UART 时间帧升级为 `time:YYYY-MM-DDTHH:MM:SS+08:00`，RA8P1 仍兼容旧 `HH:MM:SS`。
- MQTT `status/telemetry` 增加 `clock.local_iso`，Web 计算设备时间与服务器时间偏差。

## 兼容边界

- AHT20、BH1750 和 SG90 通过能力名解耦：`env.temperature`、`env.humidity`、`env.light.lux`、`motor.servo.angle`。
- 既有观测查询、`threshold_control`、`rule_program`、MQTT Topic、签名和设备执行回执均保留。
- 模型负责理解和选择动作类型，服务端负责数值范围、设备、方向和任务类型校验。
- 生产服务器上的旧 `embedded-agent-mock-device.service` 已停用，避免固定模拟值污染真实数据和误触发规则。

## 2026-06-23 上下文与定时消息补强

- 任务澄清后的短句会继承同一对话中的任务主题，例如“今天；改成 10 点 25”可更新为一次性环境上报，不要求再次写出“温湿度和光照”。
- “改成每天 / 仅今天 / 取消刚才那个”等表达会优先作用于当前对话最近的相关任务。
- 调度器产生的主动上报仍写入原 Web 对话；网页每 6 秒同步一次活动对话，因此无需手动刷新才能看到定时消息。
- 普通追问不再强制进入硬件结构化决策；Hermes 输出格式异常或上游模型不可用时，服务端会依据对话、任务状态和真实上报数据给出安全回答，不向用户暴露内部 JSON/HTTP 错误。
- Hermes 的任务上下文包含北京时间完整 ISO 时间、年月日、时分秒和星期；“今天 / 明天 / 后天”均基于 `Asia/Shanghai` 解析。
- 一次性任务保存 `target_local_iso`，支持 `十点52分`、`10点52分30秒`、`2026年6月24日十点52分30秒` 等混合表达。
- 定时汇报按用户明确指定的传感器能力读取；“上报温湿度”仅读取 AHT20 温度与湿度并主动写回创建任务的原对话。
