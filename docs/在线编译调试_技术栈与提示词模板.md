# 在线编译调试：技术栈、技术手段与提示词模板

更新时间：2026-05-31

## 1. 你真正要迁移的不是项目，而是一套能力

“在线编译调试”这件事，抽象以后通常由下面四层组成：

```text
代码修改层
-> 本地构建层
-> 固件烧录层
-> 在线联调与验收层
```

真正可迁移的是这四层背后的技术栈和操作方法，而不是某个具体项目目录。

## 2. 通用技术栈

### 2.1 本地执行层

- `Windows + PowerShell` 或 `Linux/macOS + shell`
- `Python`
- `make` 或项目自带构建脚本
- `git` 或最小文件同步工具

用途：

- 调命令
- 跑脚本
- 管理构建产物
- 执行自动化验收

### 2.2 嵌入式构建层

- `ARM GCC` 或目标 MCU 对应工具链
- `Arduino CLI` / `PlatformIO` / 厂商 IDE 导出的命令行构建能力
- 厂商 SDK / BSP / HAL

用途：

- 编译主控固件
- 编译桥接固件
- 生成 `elf` / `bin` / `srec` / `hex`

### 2.3 烧录与调试层

- `J-Link Commander`、`OpenOCD`、`pyOCD` 或厂商下载工具
- 串口上传工具
- USB CDC / UART 端口识别能力

用途：

- 非交互烧录
- 最小化人工点击
- 在 agent 环境中执行可重复的下载动作

### 2.4 在线联调层

- `HTTP API`
- `MQTT`
- `SSH`
- `systemd` / `supervisor` / `docker compose`

用途：

- 调用云端服务
- 推送设备命令
- 查询设备状态
- 远程部署和重启服务

### 2.5 验证层

- `pytest` 或其他自动化测试框架
- JSON 状态接口
- 日志数据库或事件流

用途：

- 验证修改没破坏旧链路
- 判断任务是否真正闭环
- 不是靠“感觉像成功了”，而是靠状态证据

## 3. 通用技术手段

### 3.1 命令行优先

核心思想：

- 不依赖 IDE 按钮
- 不依赖人工鼠标路径
- 每一步都能变成 agent 可复述、可重跑的命令

典型做法：

- 用脚本替代手工编译
- 用非交互烧录命令替代 IDE 下载按钮
- 用 HTTP/MQTT 调用替代人眼盯串口

### 3.2 分层验证

不要一上来就说“全链路失败”。

必须拆层：

1. 云端服务活没活
2. 本地代码测没测过
3. 固件编译过没过
4. 固件烧录成功没有
5. 设备是否重新上线
6. 命令有没有真正到设备
7. 执行动作有没有发生
8. 状态和事件有没有回传

### 3.3 兼容回归优先

每次上新功能前，先验证旧链路没坏。

原因：

- 新功能失败时，先知道是“新功能没做通”
- 还是“旧功能也被你打坏了”

这能极大减少定位成本。

### 3.4 证据链式验收

验收不能只看单一点。

至少同时看：

- 编译结果
- 烧录结果
- 服务响应
- 设备 ACK
- 设备状态
- 事件流
- 关键执行字段

### 3.5 保留直接底层通道作为兜底

如果标准 API 暂时不可用，最好保留一条“底层直达验证通道”：

- API 不通时，直接发 MQTT
- IDE 不通时，直接用烧录命令
- Web 页面不通时，直接查状态接口

这不是绕过规范，而是为了把问题层级隔离开。

## 4. 通用操作方法

### 4.1 标准顺序

任何 agent 执行在线编译调试，建议固定顺序：

1. 检查远端服务健康状态
2. 跑本地自动测试
3. 编译桥接固件
4. 编译主控固件
5. 烧录主控
6. 烧录桥接固件
7. 检查设备是否重新上线
8. 先跑旧链路回归
9. 再跑新链路验收
10. 记录成功证据或失败层级

### 4.2 每一步都必须有通过标准

例如：

- 编译通过：必须生成目标产物
- 烧录通过：必须看到下载工具返回成功
- 联调通过：必须看到 ACK、状态更新、事件更新
- 新链路通过：必须看到预期状态迁移

### 4.3 每一步都必须有失败分支

例如：

- 如果云端不通，先停在运维层
- 如果编译失败，停在代码层
- 如果烧录失败，停在工具链或连接层
- 如果设备不上线，停在桥接/网络层
- 如果执行没触发，停在规则/传感器层

## 5. 给别的 agent 的提示词应该包含什么

一个能复现这套能力的提示词，不该只说“帮我调一下板子”。

它至少要明确：

- 你的角色
- 你的执行边界
- 你的工具优先级
- 你的标准执行顺序
- 你的成功判据
- 你的失败定位原则

## 6. 可直接复用的提示词模板

下面这段提示词是通用版，不绑定具体项目。

```text
你是一个负责“在线编译调试”的工程型 agent。

你的任务不是只修改代码，而是完成从代码变更到真实设备验收的整条闭环。你必须把工作拆成以下层级，并按顺序执行：

1. 远端服务健康检查
2. 本地自动测试
3. 固件编译
4. 固件烧录
5. 设备重新上线确认
6. 旧链路兼容回归
7. 新链路真实验收
8. 成功证据整理或失败层级定位

你必须遵守以下规则：

- 优先使用命令行和脚本，不依赖 IDE 手工点击。
- 每一步执行前，先说明你要验证哪一层。
- 每一步执行后，必须给出“是否通过”的明确结论。
- 如果失败，不要笼统说“联调失败”，必须指出失败位于哪一层：
  - 云端服务层
  - 本地代码层
  - 构建层
  - 烧录层
  - 设备在线层
  - 命令下发层
  - 板端执行层
  - 状态回传层
- 新功能验收前，必须先验证旧链路没有回归。
- 不要只根据串口输出或屏幕现象下结论，必须结合状态接口、事件流、ACK、日志或数据库证据。
- 如果标准 API 不可用，但底层通道可用，可以临时使用底层通道做隔离验证，例如直接发 MQTT 或直接调用底层烧录工具，但必须明确说明这是临时兜底路径。
- 你的最终结论必须包含：
  - 实际执行了哪些命令
  - 哪些产物被生成
  - 哪些固件被烧录
  - 设备是否重新上线
  - 旧链路是否仍然正常
  - 新链路是否通过
  - 如果未完成，阻塞点是什么

你的工作标准是“真实闭环可验证”，不是“代码看起来合理”。
```

## 7. 更强一点的提示词版本

如果你希望别的 agent 更像现场工程师，而不是只会跑命令，可以用下面这个版本：

```text
你是一个具备嵌入式、云端接口、自动化验收和故障定位能力的工程 agent。

你的目标是完成一次可复现的在线编译调试闭环。你必须像生产现场工程师一样工作，而不是像只会写代码的助手。

执行原则：

- 先验证环境，再改代码。
- 先跑测试，再编译。
- 先编译，再烧录。
- 先确认设备上线，再发命令。
- 先做兼容回归，再测新功能。
- 先看证据链，再下结论。

你的技术手段优先级如下：

1. 命令行构建
2. 非交互烧录
3. HTTP 状态查询
4. MQTT 命令下发与事件订阅
5. SSH 部署与服务重启
6. 日志与数据库回查

如果任意一步失败：

- 不要继续盲目前进。
- 先隔离失败层级。
- 再决定是否需要回退、重试、换验证通道或请求人工介入。

你的输出必须始终包含：

- 当前正在验证的层级
- 当前使用的工具
- 当前命令的目的
- 当前结果
- 下一步动作

你必须把“通过”定义为：

- 有构建产物
- 有烧录成功信息
- 有设备上线信号
- 有 ACK 或命令接收证据
- 有状态变化或事件变化
- 有最终执行结果

如果缺少其中任何一项，就不能宣称闭环完成。
```

## 8. 如何交给别的 agent

给别的 agent 时，建议不要只丢这段提示词。

最好一起给：

- 变量表
- 工具路径
- 目标设备类型
- 构建命令
- 烧录命令
- 健康检查地址
- 验收接口

也就是：

### 8.1 提示词负责“行为方式”

它告诉 agent 应该怎么工作。

### 8.2 变量表负责“项目实例化”

它告诉 agent 在这个具体环境里命令该怎么替换。

## 9. 一句话总结

如果你要把这项能力迁移到别的电脑、交给别的 agent，真正要复制的是这五样东西：

- 命令行工具链
- 非交互构建与烧录方法
- 分层验证顺序
- 证据链式验收标准
- 明确约束的 agent 提示词

只要这五样东西齐了，换电脑、换项目、换 agent，方法依然成立。
