# PCA9548A + AHT20 + BH1750 联调测试与验收标准

## 1. 文档目的

本文件用于当前真实硬件接线方案下的联调验收。

目标不是“看起来差不多能用”，而是给测试人员一个可执行、可观察、可判定的标准：

- 怎么接线
- 上电后先看什么
- 屏幕上看到什么算成功
- 网页上看到什么算成功
- 哪些现象不能算通过

## 2. 当前测试范围

本次测试只针对以下现状：

- `RA8P1` 已接入最新主固件
- `ESP32-S3` 已接入最新桥接固件
- `PCA9548A` 用于复用 `I2C-1 / Bus S1 / P309+P306`
- `AHT20` 挂在 `PCA9548A CH0`
- `BH1750` 挂在 `PCA9548A CH1`

本次测试验证的是：

1. `I2C-1` 单物理标准口下多模块接入是否成功
2. `AHT20` 温湿度能力是否继续正常
3. `BH1750` 光照能力是否进入统一状态链路
4. 屏幕、ESP32 上报、网页状态是否一致或可解释一致

## 3. 当前实现边界

这一条必须先说清楚，否则测试会误判。

### 3.1 屏幕当前按动态模块入口显示，不做固定 BH1750 专页

当前屏幕 UI 已改为从统一状态动态生成模块入口。

这意味着：

- 插上什么模块，就显示什么模块
- 同一标准口下如果有多个模块，就显示多个入口
- `BH1750` 不应以“写死的独立页面”出现
- `BH1750` 应作为 `i2c.s1` 下的一个真实已接入模块动态出现

所以本次屏幕侧应该同时承担两类验证：

- 动态入口是否按真实接入结果生成
- 各入口里的状态和值是否与网页一致

### 3.2 本次不允许把 BH1750 伪装成 I2C-2

当前 `BH1750` 虽然通过 `PCA9548A` 分到子通道，但物理上仍属于：

- `i2c.s1`
- `I2C-1`
- `Bus S1`

所以测试通过时应看到：

- `i2c.s1` 下具备多个能力

而不是：

- `i2c.s2` 被错误激活

## 4. 接线标准

### 4.1 RA8P1 到 PCA9548A

| RA8P1 扩展板 | PCA9548A | 说明 |
|---|---|---|
| `P309` | `SDA` | `I2C-1` 数据 |
| `P306` | `SCL` | `I2C-1` 时钟 |
| `3V3` | `VIN` | 3.3V 供电 |
| `GND` | `GND` | 共地 |

### 4.2 PCA9548A 到 AHT20

| PCA9548A | AHT20 | 说明 |
|---|---|---|
| `SD0` | `SDA` | 子通道 0 数据 |
| `SC0` | `SCL` | 子通道 0 时钟 |
| `3V3` | `VCC` | 3.3V 供电 |
| `GND` | `GND` | 共地 |

### 4.3 PCA9548A 到 BH1750

| PCA9548A | BH1750 | 说明 |
|---|---|---|
| `SD1` | `SDA` | 子通道 1 数据 |
| `SC1` | `SCL` | 子通道 1 时钟 |
| `3V3` | `VCC` | 3.3V 供电 |
| `GND` | `GND` | 共地 |

### 4.4 本次接线约束

- `AHT20` 不能再直接挂在 `RA8P1 P309/P306` 上
- `BH1750` 也不能直接和 `AHT20` 并到 `P309/P306`
- `AHT20` 与 `BH1750` 都必须挂在 `PCA9548A` 下游子通道
- 本次默认：
  - `AHT20 -> CH0`
  - `BH1750 -> CH1`

## 6. 观察入口

测试时必须同时看两个地方：

### 6.1 屏幕侧

用于观察：

- “接入硬件”页是否动态出现多个模块入口
- `AHT20` 是否在线
- `BH1750` 是否在线
- 温湿度与光照数值是否正常
- `I2C-1` 扫描结果是否含 `PCA9548A + AHT20 + BH1750`

### 6.2 网页侧

用于观察：

- `status.ports`
- `telemetry.samples`
- `i2c.s1` 的能力是否完整
- `env.light.lux` 是否真的上来

如果网页普通设备页没有把字段完全展开，则必须进入网页的设备诊断页、原始状态页或能看到 `ports/samples` 的页面。

## 7. 测试步骤

## 7.1 步骤 A：上电基础检查

上电后等待 `20` 秒内，确认：

1. 屏幕正常亮起
2. ESP32 显示在线或 WiFi/MQTT 正常
3. 网页能看到该设备在线

### 步骤 A 通过条件

- 三项都满足

### 步骤 A 失败条件

- 屏幕不启动
- ESP32/WiFi/MQTT 未上线
- 网页设备完全无刷新

## 7.2 步骤 B：屏幕侧 I2C 扫描检查

在屏幕的 `I2C-1` 或硬件扫描信息处，观察扫描结果。

### 预期扫描结果

在当前只有这三个设备参与时，扫描结果应包含以下三项：

- `0x70/9548A-MUX`
- `0x38/AHT20`
- `0x23/BH1750`

如果 `BH1750` 模块地址脚状态不同，也可能是：

- `0x5C/BH1750`

### 屏幕动态入口预期

在“接入硬件”页面，应至少看到以下动态入口：

- `温湿度模块`
- `光照模块`

副信息可以带具体型号，例如：

- `AHT20`
- `BH1750`

### 步骤 B 通过条件

满足以下全部条件才算通过：

1. 能看到 `0x70/9548A-MUX`
2. 能看到 `AHT20`
3. 能看到 `BH1750`
4. 三者都在 `I2C-1` 的同一组扫描结果里
5. 屏幕“接入硬件”页动态出现 `AHT20/BH1750` 两个模块入口，而不是只显示一个固定 I2C 页面

### 步骤 B 失败条件

- 只看到 `AHT20`，看不到 `9548A-MUX`
- 只看到 `9548A-MUX`，看不到下游模块
- 看到 `AHT20`，但看不到 `BH1750`
- `BH1750` 出现在别的假口径里，而不是 `I2C-1`
- 屏幕只保留一个固定硬件页，没有按当前接入结果生成多个入口

## 7.3 步骤 C：屏幕侧 AHT20 功能检查

观察屏幕上的 `AHT20` 状态和温湿度读数。

### 预期现象

- `AHT20` 入口状态为 `online`
- `BH1750` 入口状态为 `online`
- `AHT20` 页面或详情中温度和湿度不是空值
- `BH1750` 页面或详情中 `lux` 不是空值
- 不是错误文本，例如：
  - `offline`
  - `write addr nack`
  - `read addr nack`
  - `sda stuck`
  - `scl stuck`

### 指标

- 上电后 `20` 秒内应出现有效温湿度和有效 `lux`
- 后续观测 `10` 秒内，不应反复跳到离线错误

### 步骤 C 通过条件

- `AHT20` 保持在线
- `BH1750` 保持在线
- 温湿度值持续有效
- `lux` 值持续有效

### 步骤 C 失败条件

- `AHT20` 离线
- `BH1750` 离线
- 温湿度显示错误诊断
- 光照显示错误诊断
- 接入 `PCA9548A` 后 AHT20 功能退化

## 7.4 步骤 D：网页侧标准口结构检查

在网页设备页、诊断页或原始状态页观察 `status.ports`。

### 预期结构

必须满足以下事实：

1. `i2c.s1` 仍是 `I2C-1 / Bus S1`
2. `i2c.s2` 仍是 `reserved`
3. `BH1750` 不能被错误显示成 `i2c.s2`

### 当前预期口径

在当前实现下，`i2c.s1` 的模块描述应表现为一类“复用后的总线入口”，重点不是给它伪造第二物理口，而是让它具备多能力。

因此网页侧至少应满足以下之一：

1. `i2c.s1.module.module_type = 9548A-MUX`
2. 或网页等价地展示 `i2c.s1` 已承载多个 I2C 子模块

### 步骤 D 通过条件

满足以下全部条件：

1. `i2c.s1` 被正确识别为当前唯一物理 I2C 标准口
2. `i2c.s2` 未被错误激活
3. 网页没有把 `BH1750` 映射成假的第二 I2C 物理口

### 步骤 D 失败条件

- 网页显示 `i2c.s2 online`
- 网页把 `BH1750` 归到 `i2c.s2`
- 网页完全看不到 `i2c.s1` 的多模块状态

## 7.5 步骤 E：网页侧 capability 检查

观察 `i2c.s1` 对应的能力列表。

### 预期 capability

当前接线下，`i2c.s1` 必须至少具备：

- `env.temperature`
- `env.humidity`
- `env.light.lux`

### 步骤 E 通过条件

- 三个 capability 同时存在

### 步骤 E 失败条件

- 只有温湿度，没有光照
- 只有光照，没有温湿度
- 仍然只按旧模块专用字段显示，看不到 capability 层

## 7.6 步骤 F：网页侧实时样本检查

观察 `telemetry.samples` 或网页实时数值区。

### 预期样本

至少要有以下三类样本：

- `AHT20 -> env.temperature`
- `AHT20 -> env.humidity`
- `BH1750 -> env.light.lux`

### 指标

- 上电后 `20` 秒内，应能看到 `env.light.lux`
- `env.light.lux` 必须是有效数值，不能长期缺失

### 步骤 F 通过条件

- 三类样本都出现
- `BH1750` 的 `env.light.lux` 为有效数值

### 步骤 F 失败条件

- `env.light.lux` 不存在
- `env.light.lux` 始终为空
- 只有扫描识别到 `BH1750`，但没有实际 `lux` 数据

## 7.7 步骤 G：BH1750 动态响应检查

在网页侧观察 `env.light.lux`，做两次环境变化：

1. 用手完全遮挡 `BH1750`
2. 用手机手电筒或强光照射 `BH1750`

### 预期现象

- 遮挡后 `lux` 明显下降
- 照射后 `lux` 明显上升

### 建议判定指标

在当前室内环境下，满足以下任一项即可视为“动态响应通过”：

1. 遮挡与照射两种状态之间，`lux` 差值大于等于 `50`
2. 或相对变化幅度大于等于 `30%`

### 时间指标

- 每次环境变化后，`15` 秒内网页应出现新数值变化

### 步骤 G 通过条件

- `lux` 对遮挡和照射有明显方向性变化

### 步骤 G 失败条件

- `lux` 长时间固定不变
- 遮挡和照射后无明显变化
- 网页只显示一次旧值，之后不刷新

## 10. 常见失败现象与定位方向

### 10.1 屏幕只显示 `0x70/9548A-MUX`

说明：

- `PCA9548A` 本体接通了
- 下游 `AHT20/BH1750` 子通道没有正常被扫到

优先排查：

- `SD0/SC0` 是否接对 AHT20
- `SD1/SC1` 是否接对 BH1750
- 两个模块的 `3V3/GND` 是否真的接上

### 10.2 AHT20 掉线，BH1750 也没上来

说明：

- 多数情况下是 `PCA9548A` 接线或供电问题

优先排查：

- `RA8P1 -> PCA9548A` 的 `SDA/SCL/VIN/GND`
- 是否错误用了 `5V`
- 是否漏接共地

### 10.3 网页没有 `env.light.lux`

说明：

- 可能板端没读到 BH1750
- 也可能 ESP32 上报链路或网页消费层未刷新

先看：

1. 屏幕扫描里是否已经看到 `BH1750`
2. 网页原始 `samples` 是否存在 `env.light.lux`
3. 如果原始状态已有而页面没显示，是网页消费层问题，不是板端问题

### 10.4 网页把 BH1750 显示成 `i2c.s2`

说明：

- 这是状态模型错误
- 本次直接判定为不通过

因为这违反了当前实现的真实物理口语义。

## 11. 当前版本的正确理解

这次通过后的正确产品语义是：

- 一个真实物理标准口：`i2c.s1`
- 一个复用器入口：`PCA9548A`
- 两个当前挂载模块：
  - `AHT20`
  - `BH1750`
- 三个当前可用能力：
  - `env.temperature`
  - `env.humidity`
  - `env.light.lux`

不是：

- `AHT20` 还在直连口
- `BH1750` 被当成第二物理 I2C 口
- 屏幕为了 `BH1750` 单独硬编码一个固定页面
