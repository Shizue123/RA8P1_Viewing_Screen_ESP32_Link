# 即插即用模块识别与激活规则 V1

## 目标

第一版不是做“完全自动识别一切”的神话方案，而是做`受控型即插即用`：

- 能自动识别到能力层的，先激活通用能力
- 能自动识别到具体型号的，直接落到具体模块
- 识别不了的，由用户在网页确认
- 屏幕和网页只面向`模块`和`功能`
- 板端、ESP32、服务器内部保留`端口 / 探测特征 / 绑定来源`

## 面向用户的展示原则

- 主标题显示模块能力层
  - 例如：`温湿度模块`
- 副信息显示型号层
  - 例如：`AHT20`、`SHT30`
- 未确认型号时：
  - 主标题仍显示能力层
  - 副信息显示：`型号待确认`

用户界面不默认暴露：

- GPIO
- SDA / SCL
- 地址
- 通道号
- capability 原始字段

这些字段只作为内部标准化数据使用。

## V1 真源划分

### 1. 板端真源：硬件事实

- 哪个标准口在线
- 最近一次探测结果
- 最近一次采样结果
- 当前诊断状态

### 2. 服务器真源：配置事实

- 用户确认了什么模块
- 页面该开放什么入口
- 对话该默认作用于哪个模块
- 20 秒自动继承窗口

### 3. 网页 / 屏幕：消费层

- 展示板端实时状态
- 展示服务器保存的模块定义
- 发生冲突时显式提示，不私自猜测

## 标准口原则

### I2C

- 正式支持`一个标准口下可存在多个模块`的产品方向
- 但当前固件阶段仍先以“单个当前活动模块快照”上报为主
- 后续接入 `TCA9548A` 时，再扩展成子通道/多模块结构

### GPIO / PWM

- 当前没有硬件检测机制
- 默认不做“物理已插入”的自动判断
- 由用户在网页手动确认已接入

## V1 状态机

### 通道未激活

- 未探测到设备
- 或该口尚未被用户启用

### 通道已激活

- 探测到口上有设备
- 但当前只能确定“口上有东西”，不能开放通用功能

### 通用模块可用

- 已确定能力层
- 可以开放基础读数或基础控制

### 具体型号待确认

- 能力层成立
- 但型号层不稳定或未确认

### 具体型号已确认

- 自动精确命中白名单
- 或用户手动确认

### 疑似模块变更

- 同一位置重新探测到设备
- 特征和历史不一致
- 系统只提示`可能已更换模块`

### 已断开

- 连续 `2-5 秒` 探测不到

## V1 白名单策略

第一版只对白名单承诺完整体验：

- `AHT20`
- `SHT30`
- `BH1750`
- `有源蜂鸣器`

白名单外的同信道器件：

- 允许检测
- 允许提示
- 允许备案
- 不承诺直接可用

## V1 关键规则

### 1. 能力层优先于型号层

例如：

- `AHT20`
- `SHT30`

都可归到`温湿度模块`。

### 2. 同类模块更换只做“概率判断”

很多低成本 I2C 模块没有唯一硬件 ID。  
系统通常只能根据以下信息判断：

- 标准口
- 地址
- 探测响应
- 驱动命中结果

因此第一版只能输出：

- `可能已更换模块`

不能虚构成：

- `100% 确认为新物理模块`

### 3. 多同类模块场景的默认对象

- 如果当前网页已进入某个模块详情页
- 对话或控制默认作用于当前模块
- 但必须提醒用户当前作用对象

### 4. 自动继承窗口

- 用户确认结果短时自动继承窗口：`20 秒`
- 超过窗口或特征变化，则重新提示确认

## 当前固件阶段已落地的最小实现

当前代码先落地这几个内部字段，作为后续扩展基座：

- `activation`
- `module.module_class`
- `module.model_state`
- `module.binding_source`
- `module.device_key`

当前这些字段的含义是：

- `activation`
  - `inactive`
  - `channel_active`
  - `confirmed`
  - `reserved`
- `module_class`
  - 能力层或模块类别
  - 例如：`env.th`、`act.servo`
- `model_state`
  - `exact`
  - `candidate`
  - `unknown`
- `binding_source`
  - `auto_exact`
  - `auto_detected`
  - `user_confirmed`
  - `system_fixed`
- `device_key`
  - 当前可见探测特征拼接键
  - 不是全局唯一硬件 ID

## 当前明确不做的事情

- 不把 GPIO/PWM 假装成能自动识别硬件
- 不把低成本 I2C 模块假装成有唯一硬件身份证
- 不在第一版直接重写成完整“一口多模块子设备树”
- 不让网页、屏幕、后端各自私造一套字段
